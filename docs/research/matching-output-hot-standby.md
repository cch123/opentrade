# 撮合输出热备仲裁架构参考

> 本文是外部撮合系统架构讨论的抽象整理，用于给本项目的撮合高可用、输出补洞、下游连续消费设计提供参考。文中不使用厂商名、内部服务名或具体 topic 名；它不是 ADR，不代表本项目已采纳该实现。

## 1. 核心问题

撮合系统的难点不只是让某个交易对的订单按顺序进入撮合状态机，还包括让下游只看到一条可信、连续、可恢复的成交与订单回报流。

如果撮合输出只有单路，一旦该路生产者、消息队列、消费者或网络链路出现短暂故障，下游可能遇到：

- 回报序号断档；
- 成交回报延迟或丢失；
- 重启后重复消费；
- 切换主备时旧消息晚到；
- 下游业务状态和撮合状态不一致。

因此，高可用设计通常会把“撮合执行”和“输出给下游的可信流”拆成两个层次：

```text
订单请求流
  -> 按交易对分片的撮合状态机
  -> 双路撮合输出流
  -> 输出仲裁与补洞层
  -> 重新编号后的下游回报流
  -> 下游业务系统 / 推送 / 历史投影
```

撮合层可以有多个活跃实例，但下游不直接消费任意一路撮合输出。下游只消费仲裁层产出的单一连续流。

## 2. 请求侧分片

订单请求按交易对维度进入不同请求分片：

```text
request stream keyed by symbol
```

这个分片方式的目标是把同一个交易对的撮合输入固定到同一条有序流里。撮合状态机可以按交易对单线程处理，避免同一本 order book 上的并发写入和锁竞争。

关键约束：

- 同一交易对内必须保持输入顺序。
- 不同交易对可以并行撮合。
- 账户、资金冻结、风控等跨交易对状态不能只依赖撮合输入顺序，需要在更上游或旁路服务中解决。
- 热门交易对可能成为单分片瓶颈，后续拆分会显著提高顺序与一致性复杂度。

## 3. 双路输出

撮合层部署为双路热备输出。两个实例都可以在线运行，并分别写入自己的输出队列：

```text
             -> output.a
matching A -

request stream

matching B -
             -> output.b
```

这里的“双路”重点不是让下游同时相信两套结果，而是让系统在一路输出缺失、延迟或节点故障时，还有另一条可用于补洞和恢复的来源。

生产级实现通常需要保证：

- 两路输出带有同一个原始撮合序号；
- 同一序号对应的业务语义必须一致；
- 输出消息具备幂等标识，例如订单号、成交号、原始序号、分片号；
- 每一路输出都能被独立消费、回放和排查；
- 实例恢复后不能直接成为可信输出源，必须先追平和校验。

## 4. 输出仲裁层

双路输出之后需要一个仲裁层。它订阅两路撮合输出，选择一条主消费路径，同时在发现序号缺口时从另一条路径同步补洞。

抽象链路如下：

```text
output.a  --\
            \
             output arbiter -> filtered output -> downstream
            /
output.b  --/
```

仲裁层的主要职责：

- 根据分片配置或主备指令选择优先消费路径；
- 检查原始撮合序号是否连续；
- 当优先路径断档时，到备用路径查找缺失序号；
- 对重复消息做幂等过滤；
- 对下游需要的分片重新过滤；
- 生成下游专用的连续输出序号；
- 在两路都无法补齐缺口时报警并停止自动推进。

这层是下游可信流的出口。下游不需要理解撮合双路切换，只需要消费仲裁层产出的连续回报流。

## 5. 两级序号

这个架构通常会出现两类序号：

| 序号 | 产生方 | 作用 |
|---|---|---|
| 原始撮合序号 | 撮合状态机 | 描述撮合输出的真实顺序，用于补洞、去重、对账 |
| 下游输出序号 | 仲裁层 | 描述下游可见回报流的连续顺序，用于业务消费和恢复 |

不能把这两个序号混为一谈。

原始撮合序号关注“撮合发生了什么”，下游输出序号关注“业务侧看到了什么”。仲裁层可以过滤掉某些不属于当前下游分片的数据，也可以在双路切换后重新组织输出，因此需要重新编号。

下游消费时应检查下游输出序号连续性；仲裁层内部则检查原始撮合序号连续性。

## 6. 缺口处理

仲裁层消费优先路径时，如果发现原始撮合序号不连续，应进入补洞流程：

```text
last_seen = 100
next message = 103

missing: 101, 102
```

处理步骤：

1. 暂停向下游推进 103。
2. 到备用路径按原始撮合序号查找 101、102。
3. 找到后按 101、102、103 的顺序输出。
4. 对输出消息分配新的下游输出序号。
5. 如果两路都找不到缺失消息，报警并等待人工处理。

这个策略说明仲裁层不是简单的“选一路”，而是“主路优先，备路补洞”。它可以掩盖单路输出延迟或短暂缺失，但不能凭空修复两路都不存在的撮合结果。

## 7. 下游定位

下游有时需要根据原始撮合序号快速定位对应的下游输出序号。为此可以维护映射：

```text
original_seq -> output_seq
```

该映射可以由仲裁层写入，也可以由一个轻量索引服务维护。它的价值是：

- 下游能从某个原始撮合位点恢复；
- 排障时能从撮合输出快速定位下游回报；
- 主备切换后可以判断某个撮合结果是否已经对下游可见；
- 避免下游直接扫描大量回报流。

映射必须和仲裁层输出保持一致。否则下游可能定位到错误的回报位置。

## 8. 节点故障与恢复

当一路撮合节点故障时，仲裁层应将该路标为不可用，只从健康路径消费；如果健康路径后续也出现缺口且备用路径不可用，就只能报警。

故障节点恢复后，不应直接重新纳入可信输出。恢复流程应至少包含：

```text
DOWN
  -> RECOVERING
  -> REPLAYING
  -> CATCHING_UP
  -> SHADOW_COMPARE
  -> STANDBY_READY
  -> ACTIVE_CANDIDATE
  -> ACTIVE
```

恢复节点需要完成：

- 从本地状态、请求日志或健康节点状态重建 order book；
- 从明确的位点回放请求或内部日志；
- 追平到当前原始撮合序号附近；
- 比对 order book、open orders、最近输出 hash、序号位置；
- 通过校验后先作为备用补洞源；
- 稳定运行后再允许切为优先路径。

如果没有快照，恢复只能依赖请求日志长时间保留和确定性回放，或者从健康节点同步状态。此时请求日志保留时间必须覆盖最长故障恢复窗口，否则故障节点无法完整恢复历史状态。

## 9. 快照不是输出仲裁的必需条件

这类架构图里不一定会展示快照。原因是输出仲裁链路解决的是“下游看到哪条连续回报流”，而快照解决的是“撮合状态机如何重启恢复”。

没有快照时，仍然可以通过以下方式恢复：

- 从请求日志的最早保留位点全量回放；
- 从本地写前日志或本地状态文件恢复后追增量；
- 从健康节点拉取状态，再从某个序号之后追增量；
- 重新以新 epoch 加入，只参与之后的热备输出，不补旧洞。

但没有快照会带来明显代价：

- 故障时间越长，回放成本越高；
- 请求日志保留不足时无法自动恢复；
- 热门交易对恢复时间不可控；
- 状态校验成本更高。

因此，输出仲裁不强制要求快照，但生产系统仍然需要某种明确的状态恢复来源。

## 10. 必要状态

仲裁层至少需要维护以下状态：

| 状态 | 用途 |
|---|---|
| 分片到优先路径的映射 | 判断默认消费哪一路 |
| 当前 epoch / term | 防止旧主晚到消息被误信 |
| 最后已处理原始撮合序号 | 检测撮合输出缺口 |
| 最后已输出下游序号 | 保证下游回报连续 |
| 原始序号到下游序号的映射 | 定位、恢复、排障 |
| 已转发消息的幂等集合 | 防止重复 ack / fill |
| 分片过滤配置 | 控制哪些回报进入当前下游流 |

其中 epoch / term 很关键。没有它，系统很难判断一条晚到消息来自旧路径还是新路径，也难以安全处理切换边界。

## 11. 风险点

### 11.1 双路状态分叉

如果两路撮合实例输入不一致、回放不确定或状态恢复不一致，同一个原始序号可能产生不同业务结果。仲裁层不能只比较序号，还需要校验关键业务内容或输出 hash。

### 11.2 旧路径晚到

切换后旧路径可能继续写入或延迟送达消息。如果没有 epoch fencing，仲裁层可能把旧消息当成当前有效消息。

### 11.3 重复输出

恢复节点回放历史时可能重新产生已经输出过的回报。仲裁层必须按原始序号、订单号、成交号做幂等过滤。

### 11.4 下游序号断档

即使原始撮合序号连续，仲裁层过滤或重编号实现错误，也可能导致下游输出序号断档。下游仍然需要独立检查输出序号连续性。

### 11.5 两路都缺失

如果两路输出都找不到某个原始序号，自动补洞无法继续。此时继续向下游推进会扩大状态分叉，应报警并人工介入。

## 12. 对本项目的参考价值

本项目当前撮合与下游业务设计已经包含按交易对定序、消息日志、快照恢复和下游投影等基础能力。这个架构对后续高可用演进的启发是：

- 撮合高可用不应只讨论谁是主节点，还要定义下游可信输出如何形成。
- 下游最好只消费一条经过仲裁的连续流，而不是直接面对多路撮合输出。
- 撮合原始序号和下游输出序号应分层建模。
- 双路热备的价值不仅是故障切换，也包括输出缺口补洞。
- 恢复节点必须先追平和校验，再重新成为可信补洞源或优先路径。
- 没有快照也能恢复，但会把压力转移到请求日志保留、确定性回放和跨节点状态同步上。

如果未来本项目采用类似方案，建议先把它设计成独立 ADR，并明确以下边界：

- 是否需要双路撮合都产生输出；
- 原始撮合序号的作用域是 per-symbol 还是 per-shard；
- 仲裁层自身如何高可用；
- 下游输出序号按什么维度连续；
- 两路输出不一致时是自动选择、报警停止，还是进入人工修复；
- 恢复节点的状态来源是快照、请求日志，还是健康节点同步。
