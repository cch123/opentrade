# 头部 CEX 柜台-撮合低延迟链路经验总结

> **调研日期**：2026-04-25
>
> **资料来源**：用户提供的单页架构图。本文只抽象技术方案，不保留任何内部系统名。
>
> **讨论范围**：柜台、撮合、行情、风控、回放、后交易之间的内部通信方式，重点是柜台与撮合的热路径。

## 1. 总体判断

该类架构的核心不是“用 Kafka 连接柜台和撮合”，而是把链路拆成两层：

1. **热路径**：柜台与撮合之间使用低延迟 UDP/Aeron/SBE 类链路。
2. **异步生态路径**：Kafka / DB / 对象存储 / 回放系统用于后交易、风控副本、历史查询、推送、审计、恢复。

换句话说，Kafka 在这种架构里不是下单回包和成交回流的主链路。它更像旁路分发与持久化基础设施。

典型形态：

```text
用户入口 / 内部服务
  -> 柜台
  -> 撮合
  -> 柜台 / 行情 / 风控 / 回放 / 后交易
```

其中：

```text
柜台 -> 撮合：p2p UDP + 二进制编码
撮合 -> 下游：UDP 组播增量流
恢复 / 审计：流录制回放 + DB / Kafka / 存储
```

## 2. 柜台到撮合：点对点低延迟请求

柜台向撮合发起的请求包括：

- 下单
- 撤单
- 改单
- 保证金 / 风控相关的撮合前校验结果
- 其他订单控制类动作

这条链路适合走 **p2p UDP + SBE / Protobuf-like 二进制编码**，而不是 Kafka：

```text
柜台
  -- p2p UDP / Aeron / SBE -->
撮合
```

原因：

- 请求非常短，适合固定 schema 的二进制编码。
- 柜台和撮合都在同机房或低延迟网络内，UDP/Aeron 的收益明显。
- 下单 ACK / reject 对延迟敏感，不适合被 Kafka append、事务、consumer poll 引入毫秒级尾延迟。
- 撮合热路径通常是 per-symbol / per-shard 单线程顺序执行，网络层只要提供稳定低延迟的有序输入即可。

这里的“回包”不一定是传统 RPC 语义。更常见的是：

```text
请求包：柜台 -> 撮合
结果流：撮合 -> 组播流 -> 柜台消费
```

如果使用 Aeron Cluster，也可以实现 client session 级别的 ingress / egress request-response；但从这类 CEX 架构图看，撮合结果更偏向通过统一事件流回到柜台，而不是每笔请求都走同步 RPC 返回完整结果。

## 3. 撮合到柜台：组播权威事件流

撮合产出成交、订单状态、盘口增量之后，不逐个精确投递给某个柜台实例，而是发布到组播流：

```text
撮合
  -- UDP multicast / Aeron publication -->
  柜台集群
  行情服务
  风控副本
  回放服务
  后交易服务
```

对柜台而言，这相当于广播消费：

```text
for event in 撮合事件流:
    if event 关联的 user/order 属于本柜台分片:
        清算、改订单状态、释放冻结、写账
    else:
        跳过
```

这个模式和 OpenTrade 当前的 `dual-emit + user partition` 有本质差异：

| 维度 | OpenTrade 当前 | 组播事件流模式 |
|---|---|---|
| 撮合输出 | Match 为 maker/taker 分别写目标 partition | 撮合广播一份权威事件 |
| 路由责任 | 撮合侧计算 user shard | 柜台侧本地过滤 |
| 带宽 | 精确投递，1x | 广播放大，约等于订阅者数量 |
| 一致性 | Kafka 事务保证 maker/taker 双写原子 | 事件流顺序 + 消费端幂等 |
| 扩展性 | 多 Kafka 集群拆分困难 | 可按撮合 shard / symbol group 拆流 |
| 延迟 | Kafka ms 级 | UDP/Aeron 可做到更低 |

组播模式的关键是：**撮合只负责产出一份权威顺序流，下游各自过滤和投影**。

## 4. Kafka 的位置：不是热路径，而是异步旁路

这类架构仍然会大量使用 Kafka，但用途与 OpenTrade 当前热路径不同。

Kafka 适合承载：

- 后交易处理
- 跟单 / 策略 / 普通异步业务
- 风控副本的非实时链路
- 用户推送的边缘 fanout
- 跨机房或跨区域转发
- 审计与数据湖入口
- 降级后的补偿分发

Kafka 不适合直接承载：

- 下单同步回包
- 撮合到柜台的第一跳清算事件
- 行情 tick-by-tick 的最短路径

因此更合理的分层是：

```text
撮合权威流
  -> UDP/Aeron：柜台、行情、实时风控、回放
  -> Kafka bridge：后交易、历史、推送、审计、离线
```

## 5. 回放与恢复：流录制比 Kafka retention 更靠近热路径

组播 UDP 本身没有 Kafka 那种天然 retention，所以必须配一套录制回放机制。

典型做法：

```text
撮合事件流
  -> 流录制
  -> 快照
  -> replay
  -> 柜台 / 行情 / 风控恢复
```

恢复链路通常包含三类数据：

1. **内存快照**：撮合 book、柜台账户、活动订单、分片状态。
2. **增量事件流录制**：从快照点之后 replay 到最新。
3. **DB / AOF / Kafka 旁路**：用于审计、历史查询、长周期恢复和人肉排查。

关键点是：热路径恢复优先依赖“快照 + 录制事件流”，而不是从 Kafka 头开始慢慢扫。

这和 Kafka-first 架构的心智不同：

- Kafka-first：Kafka topic 是主要 source of truth。
- Aeron/UDP-first：撮合/柜台内存态 + 录制事件流是热路径 truth，Kafka 是外部投影与长期留存。

## 6. 柜台职责会更重

组播模式把一部分复杂度从撮合侧转移到柜台侧。

柜台必须处理：

- 本地过滤：只应用属于自己用户或订单的事件。
- 幂等：按 `(symbol, seq)` 或 `(user, symbol, seq)` 去重。
- 乱序 / gap 检测：组播流需要明确的 seq 和恢复协议。
- 双边成交：maker/taker 可能落在不同柜台分片，各分片独立结算自己的一侧。
- self-trade：maker/taker 同用户时必须在同一个用户序列化上下文里合并处理。
- 重启 catch-up：从快照点 replay 到最新，再接实时流。

这要求柜台内部也采用严格的单用户或单账户串行化模型。否则广播流并发落地时很容易把订单状态、冻结资金、成交结算打乱。

## 7. 为什么这种架构适合头部 CEX

它解决的是几个 Kafka dual-emit 模式的长期瓶颈：

1. **低延迟**：下单和撮合结果不经过 Kafka consumer poll。
2. **多下游 fanout**：行情、风控、柜台、回放天然订阅同一份权威流。
3. **多撮合集群拆分**：可以按 symbol group / 产品线拆不同撮合流，不要求跨 Kafka cluster 事务。
4. **撮合侧简单**：撮合不需要知道柜台 shard 拓扑，也不负责 maker/taker 精确投递。
5. **事件语义统一**：所有实时下游看到同一个撮合序列，减少跨系统 drift。

代价也很明确：

1. **带宽放大**：广播给多个订阅者，消费侧要过滤。
2. **运维复杂**：Aeron/UDP/组播/录制回放比 Kafka 更难调试。
3. **可靠性要自建**：gap detection、replay、快照、leader 切换、流录制都要做扎实。
4. **开发门槛高**：二进制协议、序列号、回放一致性、线程模型都需要强约束。
5. **排查成本高**：不像 Kafka 可以直接 console-consumer 看消息。

## 8. 对 OpenTrade 的启示

OpenTrade 当前保持 Kafka 模式是合理的：

```text
Counter -> Kafka order-event -> Match
Match -> Kafka trade-event -> Counter
```

这个模式简单、可观测、易回放，适合 MVP 和中等规模交易系统。短期不应该直接切到 UDP/Aeron。

但长期可以按阶段吸收经验：

### 阶段 1：保留 Kafka，先补齐事件纪律

- 明确所有撮合输出都有单调 `match_seq_id`。
- Counter 消费端继续强化 `(user, symbol, match_seq_id)` 幂等。
- Snapshot 绑定输入 offset 和输出 flush barrier。
- 明确 maker/taker 双边结算的幂等与 self-trade 合并路径。

这些能力无论未来是否切 Aeron 都必须存在。

### 阶段 2：把“撮合权威流”概念先做出来

即使底层仍是 Kafka，也可以在语义上把 `trade-event` 视为一份权威撮合流：

- 一条 TradeEvent 表达完整 maker/taker 信息。
- 下游可以选择精确 partition 消费，也可以选择广播消费后过滤。
- 行情、风控、回放都尽量围绕同一份撮合序列构建。

这会降低未来从 `dual-emit` 切到 `broadcast-filter` 的成本。

### 阶段 3：高吞吐场景评估 broadcast-filter

当单 Kafka 集群或 dual-emit 事务成为瓶颈时，优先评估：

```text
Match -> symbol/shard topic
所有 Counter 分片订阅
Counter 本地过滤
```

这仍然可以用 Kafka 实现，但语义已经接近组播模式。它是从当前架构到 UDP/Aeron 的低风险过渡。

### 阶段 4：只有在必要时引入 UDP/Aeron 热路径

触发条件应该足够硬：

- 单集群 Kafka 写入或事务协调明显成为瓶颈。
- 业务要求撮合结果回流从 ms 级压到 sub-ms。
- 团队具备 UDP/Aeron、流录制、快照回放、二进制协议运维能力。
- 已经有成熟的 gap detection / replay / 对账工具。

否则，Kafka 的可观测性和成熟度更有价值。

## 9. 推荐架构原则

如果未来要借鉴这种模式，建议坚持这些原则：

1. **热路径和生态路径分离**：低延迟链路只服务下单、撮合、清算、行情；其他业务走 Kafka / DB。
2. **撮合只产一份权威流**：不要让撮合理解所有下游拓扑。
3. **柜台必须幂等**：广播模式下重复、replay、跨分片过滤都是常态。
4. **所有事件必须有 seq**：没有序号就无法做 gap detection 和恢复。
5. **先有 replay，再谈 UDP**：没有录制回放，UDP 组播不可生产化。
6. **Kafka 仍保留**：即使用 Aeron/UDP 做热路径，也需要 Kafka 做异步分发和长期留存。
7. **不要过早替换**：只有当 Kafka 模式真的触及延迟或扩容红线时，再迁移热路径。

## 10. 一句话结论

头部 CEX 的经验不是“用某个中间件替代 Kafka”，而是：

> **撮合热路径用低延迟点对点请求 + 组播权威事件流；柜台本地过滤并幂等清算；Kafka 退到异步旁路和长期留存。**
