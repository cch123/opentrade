# ADR-0006: 快照由备节点产生

- 状态: Accepted
- 日期: 2026-04-18
- 决策者: xargin, Claude
- 相关 ADR: 0001, 0002

## 背景

Counter 和 Match 都是内存密集型有状态服务，重启时从 Kafka 头部回放全部历史事件会非常慢。需要快照（periodic snapshot）作为回放起点。

问题：快照由哪个节点产生？主节点打快照会影响请求处理延迟（迭代大 map 需要读锁或 COW fork）。

## 决策

**快照由备节点产生。** 主节点专注低延迟请求处理，备节点反正一直在 tail Kafka，顺手做快照几乎零额外代价。

## 备选方案

### 方案 A：主节点打快照
- 优点：数据最新
- 缺点：
  - 迭代大 map 要加锁 → 影响请求延迟
  - COW fork（类似 Redis BGSAVE）在 Go 里不自然
  - 占用主的 CPU/内存

### 方案 B：独立 snapshot worker
- 单独一个进程消费 Kafka，重建状态，打快照
- 等于再起一个备节点，纯冗余，还要维护

### 方案 C（选中）：备节点打快照
- 优点：
  - 备本来就在追平状态，无额外重建成本
  - 不影响主
  - 任何时刻备状态 = 某个 Kafka offset 的一致性视图
- 缺点：
  - 快照时刻的状态比主落后几 ms（无所谓，快照本来就是某个 offset 的定格）

## 理由

- 备节点有空闲资源
- 主专注低延迟是核心诉求
- 快照语义本来就是"某个 offset 时刻的 state"，备落后几毫秒不影响正确性（恢复时从 `snapshot.offset` 继续消费 Kafka，一致性有保证）

## 影响

### 正面
- 主节点延迟稳定，不会因快照波动
- 零开发成本的快照机制

### 负面
- 备节点需要预留磁盘/CPU 做快照序列化
- 若主和备同时挂，最新快照滞后于主的实际状态（但 Kafka 有完整事件，恢复时从快照继续消费即可，无数据损失）

## 实施约束

- **触发条件**：每 10,000 条事件 或 每 60 秒（取先到）
- **存储**：本地 EFS/SSD 写入 + 异步上传 S3 作为跨机房备份
- **命名**：`{service}-shard-{N}-offset-{M}-{ts}.snap`
  - 例：`counter-shard-3-offset-1234567-20260418T120000.snap`
- **格式**：protobuf-encoded state + 元数据
  - 元数据必含：`service_name, shard_id, offset_map, checksum, version`
- **保留策略**：本地最近 3 份，S3 保留 7 天
- **原子性**：写入时先 `.tmp` → fsync → rename，避免加载到不完整快照
- **加载校验**：启动时校验 checksum，失败则回退到前一份，再失败则从 Kafka 头回放
